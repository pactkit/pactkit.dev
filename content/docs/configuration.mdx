## Deployment Architecture

PactKit supports two deployment formats, each targeting the AI tool's native directory structure:

### Claude Code (Classic)

```

Ōö£ŌöĆŌöĆ commands/                 ŌåÉ 11 command playbooks
ŌööŌöĆŌöĆ skills/                   ŌåÉ 10 skills (3 scripted + 7 prompt-only)
```

### OpenCode

```

Ōö£ŌöĆŌöĆ rules/                    ŌåÉ 7 rule files (loaded via opencode.json)
Ōö£ŌöĆŌöĆ agents/                   ŌåÉ 9 agents (mode: subagent, record tools)
Ōö£ŌöĆŌöĆ commands/                 ŌåÉ 11 commands (agent: build + model: routing)
Ōö£ŌöĆŌöĆ skills/                   ŌåÉ 10 skills (SKILL.md frontmatter)
ŌööŌöĆŌöĆ opencode.json             ŌåÉ Global config (instructions, provider preserved)
```

## pactkit.yaml

The `pactkit.yaml` file controls which components are deployed and how they behave.

**Location**: PactKit auto-detects the config file based on your environment:

| Environment | Config Path |
|-------------|-------------|
| Claude Code | `.claude/pactkit.yaml` |
| OpenCode | `.opencode/pactkit.yaml` |

Both paths are checked automatically ŌĆö no manual configuration needed.

### Reference

| Field | Type | Default | Description |
|-------|------|---------|-------------|
| `stack` | string | auto-detected | Project stack (`python`, `node`, `go`, `java`) |
| `version` | string | current | PactKit version that generated the config |
| `developer` | string | `""` | Developer prefix for Story IDs (multi-developer collaboration) |
| `agents` | list | all 9 | Agent definitions to deploy |
| `commands` | list | all 11 | Command playbooks to deploy |
| `skills` | list | all 10 | Skills to deploy |
| `rules` | list | all 7 | Constitution rule modules to deploy |
| `ci` | object | `provider: none` | CI/CD pipeline generation (`github`, `gitlab`, `none`) |
| `issue_tracker` | object | `provider: none` | External issue tracker (`github`, `none`) |
| `hooks` | object | disabled | Opt-in hook templates (pre-commit, post-test, pre-push) |
| `lint_blocking` | bool | `false` | Whether lint failures block commits in Done command |
| `auto_fix` | bool | `false` | Whether to auto-fix lint errors before checking |
| `agent_models` | object | `{}` | Per-agent model overrides (`haiku`, `sonnet`, `opus`, `inherit`) |
| `command_models` | object | defaults | Per-command model overrides for OpenCode deployment |
| `check.security_checklist` | bool | `true` | Enable 8-item structured security checklist in Check phase |
| `done.lesson_quality_threshold` | int | `15` | Minimum quality score (0-25) for lessons to be auto-appended |
| `rule_scopes` | object | `{}` | Map rule IDs to glob patterns for context-aware scoping |
| `enterprise` | object | all `false` | Enterprise flags: `no_git`, `no_external`, `non_interactive`, `debug` |

<Callout type="info">
All conditional features (CI/CD, issue tracking, hooks, lint blocking) are **disabled by default** and only activate when explicitly configured.
</Callout>

### Multi-Developer Collaboration

Set `developer` to avoid Story ID conflicts across team members:

```yaml
developer: alice
```

Story IDs become `STORY-alice-001`, `STORY-alice-002`, etc. Each developer has their own ID namespace, preventing file conflicts during merge.

### Command Model Routing (OpenCode)

The `command_models` section controls which model each command uses when deployed to OpenCode:

```yaml
command_models:
  project-act: sonnet      # Implementation ŌåÆ Sonnet
  project-done: sonnet     # Commit gate ŌåÆ Sonnet
  project-check: sonnet    # QA audit ŌåÆ Sonnet
  project-init: sonnet     # Setup ŌåÆ Sonnet
  project-release: sonnet  # Release ŌåÆ Sonnet
  project-pr: sonnet       # PR creation ŌåÆ Sonnet
  project-hotfix: sonnet   # Hotfix ŌåÆ Sonnet
```

Commands not listed (e.g., `project-plan`, `project-clarify`, `project-design`) inherit the main model, so you can switch between Opus versions freely.

Model shortnames (`sonnet`, `opus`, `haiku`) are automatically resolved to full `provider/model-id` format by reading your `opencode.json` provider configuration.

## Constitution Rules

The 7 rule modules form the "constitution" that governs agent behavior:

| Module | Purpose |
|--------|---------|
| **Core Protocol** | Session Context, Visual First, Strict TDD, Language Matching |
| **Hierarchy of Truth** | Spec > Tests > Code precedence, conflict resolution rules |
| **File Atlas** | Standard file locations and their purposes |
| **Routing Table** | Maps commands to agents and playbooks |
| **Workflow Conventions** | Conventional Commits, branch naming, PR conventions |
| **MCP Integration** | Conditional MCP server usage by PDCA phase |
| **Shared Protocols** | Cross-command protocols (Lazy Visualize, Test Mapping, Context.md format) |

## Language Profiles

PactKit auto-detects the project stack and configures the appropriate test runner and lint command:

| Stack | Test Runner | Lint Command | Package File |
|-------|------------|-------------|-------------|
| `python` | `pytest tests/` | `ruff check src/ tests/` | `pyproject.toml` |
| `node` | `npx jest` | `npx eslint .` | `package.json` |
| `go` | `go test ./...` | `golangci-lint run` | `go.mod` |
| `java` | `mvn test` | `mvn checkstyle:check` | `pom.xml` |

## Safe Regression

PactKit's safe regression system prevents agents from blindly modifying pre-existing tests:

- **TDD Loop** ŌĆö Only iterates on tests created in the current story
- **Regression Check** ŌĆö Read-only gate; pre-existing test failure = STOP and report
- **Done Gate** ŌĆö Full regression by default; incremental only when ALL safety conditions are met

## Idempotent Updates

```bash
pactkit update                    # Claude Code
pactkit upgrade --format opencode # OpenCode
```

Safe to re-run at any time. Re-deploys all managed files while preserving your custom rules and user configuration.

## Preview

Preview the deployment without modifying production directories:

```bash
pactkit init -t /tmp/preview
pactkit init --format opencode -t /tmp/oc-preview
```
